```markdown
L'API Anthropic directe coûte cher pour un usage intensif.
```

#### Leçon 4 : Resource Groups et Subscriptions

```markdown
# Resource Groups et Subscriptions

## Subscriptions : comment organiser

### Pour un usage personnel / side project

Une seule subscription suffit :
- "Visual Studio Enterprise" (si tu l'as)
- "Pay-As-You-Go"
- "Free Trial"

### Pour une entreprise

Plusieurs subscriptions :
- sub-dev (développement)
- sub-staging (pré-prod)
- sub-prod (production)

Avantage : isolation des coûts et des droits.

## Resource Groups : les bonnes pratiques

### Règle d'or

> Un resource group = un projet OU un environnement

### Convention de nommage

rg-[projet]-[environnement]-[région]

Exemples :
- rg-chatbot-dev-frc (chatbot, dev, France Central)
- rg-chatbot-prod-frc
- rg-datalake-prod-weu (West Europe)

### Ce qu'il faut savoir

1. **Tout doit être dans un resource group**
   - Pas de ressource orpheline

2. **Un resource group = une région de facturation**
```

```markdown
| Rôle | Permissions |
|------|-------------|
| **Owner** | Tout faire + gérer les accès |
| **Contributor** | Tout faire sauf gérer les accès |
| **Reader** | Lecture seule |

Rôles spécifiques IA :
| Rôle | Permissions |
|------|-------------|
| **Cognitive Services User** | Utiliser les APIs |
| **Cognitive Services Contributor** | Créer et gérer les ressources |

## Bonnes pratiques

Donne uniquement les permissions nécessaires.

❌ Owner pour tout le monde
✅ Contributor pour les devs, Reader pour les observateurs

### 2. Utilise des Managed Identities

Pour tes applications, évite les clés API stockées en dur.

### 3. Stocke les secrets dans Key Vault

# Lire un secret
az keyvault secret show --vault-name mon-keyvault --name "API-KEY" --query value

### 4. Active les logs de diagnostic

Pour auditer qui fait quoi.

## Ce que tu dois retenir

1. Ne partage jamais tes clés API en clair
2. Utilise Key Vault pour les secrets
```

```markdown
## Pourquoi évaluer ?

- Mesurer la qualité des réponses
- Comparer des modèles/prompts

## Types d'évaluation

### 1. Évaluation manuelle

Tu lis les réponses et tu juges.

✅ Simple
❌ Ne scale pas

### 2. Évaluation automatique (métriques)

| Métrique | Description | Quand l'utiliser |
|----------|-------------|------------------|
| **Groundedness** | La réponse est-elle basée sur le contexte ? | RAG |
| **Relevance** | La réponse répond-elle à la question ? | Tous |
| **Coherence** | La réponse est-elle cohérente ? | Tous |
| **Fluency** | La réponse est-elle bien écrite ? | Tous |
| **Similarity** | Proximité avec une réponse attendue | QA |
```
